# MVP仕様

## ステータス

Draft 0.1。数値上限と実装技術は検証により変更される可能性があります。

## 1. 背景

ローカルAIエージェントを実行する際、モデル、ツール、権限、資源制限、承認、監査の実装が個別アプリケーションに分散しています。MVPでは、単一のLinux端末上でこれらを統一的に扱う最小ランタイムを提供します。

## 2. ゴール / 非ゴール

### ゴール

- 構造化タスクをローカルAPIへ投入できる
- タスクごとにファイル、ネットワーク、ツールの能力を制限できる
- エージェントとモデル実行の状態および資源消費を確認できる
- 高影響操作を人間の承認まで停止できる
- 重要な状態遷移と操作を監査できる

### 非ゴール

- GUI、音声UI、一般消費者向けインストーラー
- 複数端末にまたがる分散実行
- 独自カーネルおよび独自デバイスドライバ
- あらゆるモデルランタイムへの対応
- 完全自律的な権限変更

## 3. 用語

- **Task**: 目的、制約、能力、予算をまとめた実行単位
- **Agent**: Taskを計画し、許可されたToolを呼び出す実行主体
- **Capability**: 特定資源に対する操作権限
- **Budget**: Taskが消費可能な資源の上限
- **Approval**: 高影響操作に対するユーザーの一回限りの判断
- **Event**: 状態変化または監査対象操作の追記記録

## 4. 要件

### 機能要件

- **FR-001**: ランタイムはJSON形式のTaskを受け付ける
- **FR-002**: Task IDまたは冪等性キーの重複を検出する
- **FR-003**: 未指定のCapabilityは拒否する
- **FR-004**: ファイルアクセスは正規化済みパスと操作種別で評価する
- **FR-005**: ネットワークアクセスは初期設定で拒否する
- **FR-006**: 高影響操作を承認待ちに遷移させる
- **FR-007**: Taskをキャンセルし、子プロセスを終了できる
- **FR-008**: 経過時間とメモリ使用量の上限を強制する
- **FR-009**: 状態遷移とポリシー判断をEventとして保存する
- **FR-010**: モデルランタイムを共通アダプター経由で交換できる

### Task入力例

```json
{
  "idempotency_key": "repo-analysis-2026-07-19-001",
  "goal": "リポジトリを分析し、テスト失敗の原因を報告する",
  "capabilities": {
    "filesystem": [
      { "path": "/workspace/project", "access": "read" }
    ],
    "network": "deny",
    "tools": ["test_runner"]
  },
  "budget": {
    "wall_time_seconds": 1800,
    "memory_bytes": 8589934592,
    "max_parallel_agents": 2
  },
  "approval": {
    "required_for": ["filesystem.write", "git.commit", "network.egress"]
  }
}
```

### 出力の最小項目

```json
{
  "task_id": "01J...",
  "state": "succeeded",
  "result": {
    "summary": "失敗原因と再現手順を特定しました",
    "artifacts": []
  },
  "usage": {
    "wall_time_seconds": 124,
    "peak_memory_bytes": 1207959552
  },
  "event_cursor": 42
}
```

## 5. 状態と遷移

```text
submitted -> validating -> queued -> running -> succeeded
                |            |         |  |
                v            |         |  +-> waiting_approval -> running
              rejected       |         +----> failed
                             +--------------> cancelled
```

- 終端状態は `succeeded`、`failed`、`cancelled`、`rejected`
- 終端状態から実行状態へは戻さない
- 再実行は新しいTask IDを発行し、元のTask IDを参照する
- 承認には対象操作、対象資源、有効期限を含める

## 6. 境界条件と例外

- `goal`が空、空白のみ、または8,192文字を超える場合は拒否する
- 必須項目の`null`は未指定として暗黙補完せず、検証エラーにする
- 同じ冪等性キーと同じ入力は、既存Taskを返して二重実行しない
- 同じ冪等性キーで入力が異なる場合は競合エラーにする
- `max_parallel_agents`はMVPでは1以上8以下とする
- 0以下の予算、整数範囲外、未知の単位は拒否する
- 実行中に予算へ達した場合、新規ツール呼び出しを停止してTaskを失敗させる
- 部分的な成果物は失敗結果から参照可能にするが、自動公開しない
- キャンセルは冪等とし、終端状態へのキャンセル要求は現在状態を返す

### エラーコード

- `INVALID_TASK`: 入力形式または境界条件違反
- `IDEMPOTENCY_CONFLICT`: 冪等性キーと入力の不一致
- `CAPABILITY_DENIED`: 必要な能力が未許可
- `APPROVAL_EXPIRED`: 承認の有効期限切れ
- `BUDGET_EXCEEDED`: 資源上限への到達
- `RUNTIME_UNAVAILABLE`: 指定モデルまたはツールを利用不能
- `INTERNAL_ERROR`: 内部不整合。秘密情報を含まない診断IDを返す

## 7. 非機能要件

### セキュリティとプライバシー

- 外部ネットワークは初期設定で拒否する
- モデル出力をCapability変更の根拠にしない
- 秘密情報をプロンプト、Event、エラーメッセージへ平文保存しない
- Task間でContextを暗黙に共有しない
- ポリシー評価失敗時は拒否側へ倒す

### 信頼性

- ランタイム再起動後に、Taskの終端状態とEventを復元できる
- EventはTask単位で単調増加するカーソルを持つ
- Toolの再試行は、冪等性が宣言された操作に限定する

### 性能

- ポリシー評価はモデル推論を必要としない
- アイドル時にモデルを常時ロードするかは設定可能にする
- 性能目標値はベースライン計測後に確定する

### 互換性

- 最初のサポート対象をLinux x86_64とする
- ハードウェアアクセラレータなしでも、低速な機能縮退モードを許容する

## 8. 受け入れ条件

1. Given ネットワーク能力が未指定のTask、When エージェントが外部接続を要求、Then 接続は拒否されEventへ記録される。
2. Given 読み取り専用ディレクトリ、When エージェントが書き込みを要求、Then 操作は実行されず`CAPABILITY_DENIED`になる。
3. Given メモリ上限付きTask、When 使用量が上限へ到達、Then 子プロセスが停止し`BUDGET_EXCEEDED`になる。
4. Given 承認対象のコミット操作、When エージェントが実行を要求、Then Taskは`waiting_approval`となり、承認前にコミットされない。
5. Given 同一の冪等性キーと入力、When 同時に2回投入、Then 実行されるTaskは1つだけになる。
6. Given 実行中Task、When ランタイムを再起動、Then Task状態と既存Eventを取得できる。
7. Given 2種類のモデルアダプター、When 設定を切り替える、Then Task APIの形式を変更せず実行できる。

## 9. 計測・ログ・監視

- Task数と状態別件数
- 待ち時間、実行時間、承認待ち時間
- CPU時間、ピークRAM、可能な場合はGPU/VRAM使用量
- ポリシー拒否数と理由コード
- モデルおよびToolごとの失敗数
- 予算超過と強制終了数

ログはTask IDとEvent IDで関連付けます。入力本文、秘密情報、完全なモデル応答は既定の運用ログに含めません。

## 10. 互換性・移行

MVPの永続形式は安定APIとみなしません。形式変更時はスキーマバージョンを更新し、少なくとも直前バージョンからの移行手段または明示的な再初期化手順を提供します。

## 11. 代替案とトレードオフ

- **独自カーネルから開始**: 制御範囲は広い一方、ドライバ、隔離、配布の負担がMVP価値を上回るため採用しない
- **コンテナだけで実装**: 隔離には有用だが、Task、Capability、承認、モデル管理の共通意味論が不足する
- **すべてをPythonで実装**: 試作は速いが、長期常駐する権限強制層の安全性と配布性を考え、中核には採用しない
- **クラウド前提**: 高性能モデルを利用しやすいが、ローカル優先とプライバシー目標に反する
